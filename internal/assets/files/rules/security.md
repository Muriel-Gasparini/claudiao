# Rules — Security

Security is a first-class quality gate, not an afterthought. Any new or changed
code that touches user input, auth, data, external systems, or infrastructure
MUST be evaluated against this file before declaring Ready.

## Principle

- **Deny by default.** Allow lists over block lists. Explicit authorization checks at every sensitive boundary. Absence of a check is a defect.
- **Trust no input.** Validate shape, type, and range at the boundary; never trust data from a client, a queue, a file, or a third-party API.
- **Depth in layers.** Input validation does not replace parameterization, escaping, or authorization. Use all three.
- **Least privilege everywhere.** Services, DB users, IAM roles, container users, API keys. Start from zero and add only what's needed.
- **Assume breach.** Design so a single compromised component does not pwn the system. Segment, rotate, limit blast radius.

## Secrets

- **Never** commit secrets: API keys, tokens, passwords, private keys, signing keys, DB URLs with credentials, OAuth client secrets.
- **Never** log secrets (auth headers, tokens in URLs, session IDs, passwords even "hashed for demo").
- **Never** send secrets to third parties (error trackers, analytics, LLM prompts, debug dumps).
- Read secrets from environment variables, a secrets manager, or a vault. Not from config files committed to the repo.
- **Do not put secrets in URLs** (query strings leak to logs and `Referer` headers). Use headers or request body.
- If a secret leaks, **rotate first**, remove from history second. Assume compromise.
- Add pattern checks: pre-commit hooks (`gitleaks`, `trufflehog`), `.gitignore` for `.env`, `*.pem`, `credentials.*`.
- Rotate secrets on a schedule and immediately on personnel offboarding.

## Input validation

Validate at every **trust boundary**:
- HTTP handlers (body, query, headers, path params, cookies)
- Message consumers (Kafka, SQS, webhooks, pub/sub)
- File upload contents and MIME types
- CLI args and env vars that affect behavior
- Data read from external APIs

For each field check:
- Type and shape (use a schema: zod, pydantic, JSON schema, struct tags)
- Length bounds (min and max — reject unbounded strings)
- Allowed character set (regex or explicit enum; beware of regex catastrophic backtracking)
- Semantic range (dates in plausible window, IDs in owned scope)
- Encoding (reject invalid UTF-8, null bytes, BOMs where not expected)
- Reject unknown fields when deserializing bodies (`additionalProperties: false` equivalent)

Reject early with 4xx. Do not "sanitize" silently — clients must know their input was wrong.

## Authentication & sessions

### Authentication

- Passwords: **Argon2id** (preferred) or **bcrypt** cost ≥ 12. Never MD5, SHA‑1, or unsalted SHA‑256 for passwords.
- Minimum password policy: length ≥ 12, check against known-breached list (HIBP `k-anonymity` API). No arbitrary complexity rules that hurt usability.
- MFA: offer TOTP + WebAuthn. Required for admin roles. Never SMS-only for high-value accounts.
- Throttle auth endpoints (login, reset, signup) per IP **and** per account; exponential backoff on failure.
- Account enumeration: same response/time for "no such user" and "wrong password". Same for reset flows.

### Sessions & tokens

- Session IDs/tokens: generated by **cryptographic RNG**, ≥ 128 bits of entropy.
- **Rotate session ID on login and on privilege change** (prevents session fixation).
- **Invalidate server-side on logout** (revocation list, opaque tokens, short TTL + refresh).
- Idle timeout + absolute timeout. Force re-auth for sensitive ops.
- **Browser tokens belong in `httpOnly; Secure; SameSite=Lax|Strict` cookies, not `localStorage`** (localStorage is readable by any XSS).
- Bind tokens to a context where possible (IP, user agent hash) for admin sessions.

### JWT (if used)

- Verify algorithm explicitly server-side. **Reject `alg: none`**. Reject algorithm confusion (HS vs RS).
- Validate `exp`, `iat`, `nbf`, `iss`, `aud`.
- Keep tokens short-lived (≤ 15 min). Use refresh tokens with revocation for long sessions.
- **JWTs are not a substitute for sessions** when you need revocation. Use opaque session IDs + server store, or pair JWT with a revocation list.
- Rotate signing keys; support multiple active keys (`kid`) for rollover.

### OAuth / OIDC

- **PKCE required** for public clients. `S256` only.
- **`state` parameter** required on every flow (CSRF protection).
- Validate `id_token` signature, issuer, audience, nonce.
- Do not trust the UserInfo endpoint alone; verify the ID token.
- Redirect URIs are **exact match** against an allow list (no wildcard, no prefix match).
- Store refresh tokens server-side only.

## CSRF

- For cookie-authenticated state-changing endpoints (POST/PUT/PATCH/DELETE): either
  - **`SameSite=Strict`** cookies + origin check (`Origin`/`Referer`), OR
  - **CSRF token** (double-submit cookie or synchronizer token) required on every mutating request.
- Pure Bearer-token APIs (no cookie auth) are not CSRF-vulnerable; still check `Origin` for defense in depth.
- `GET` requests must be **side-effect free**. Never use GET for mutations.

## Authorization

- **Authentication** proves who the caller is. **Authorization** proves they may do this specific thing to this specific resource.
- Check authorization **at every endpoint**, including GET endpoints. Do not rely on UI hiding.
- Scope every query to the caller's tenant/user/organization. **IDOR** (`GET /users/123`) must verify ownership, not just existence.
- Admin operations require explicit role/permission checks, not "logged in".
- Prefer attribute/relationship-based checks over role-string checks (ReBAC/ABAC) when resources are shared.
- Test negative authz cases: another tenant's ID, expired role, revoked token.

## Injection

- **SQL**: always parameterized queries or a query builder. Never string concatenation with user input. Never `ORDER BY $userInput` without allow-listing the column set.
- **Command execution**: avoid `exec`, `system`, shell-out in general. When unavoidable, never pass user input as shell args. Use the language's argument-array form, not a single string.
- **Template / HTML**: auto-escape by default. Only use `raw` / `html-safe` / `dangerouslySetInnerHTML` on content you generated yourself or sanitized with a vetted library (DOMPurify, bleach).
- **NoSQL**: reject operators from user input (`$gt`, `$ne` in Mongo). Validate that fields receive primitives, not objects.
- **LDAP, XPath**: escape per the target's rules or use parameterized APIs.
- **XXE (XML External Entity)**: disable external entities and DTDs in every XML parser (`libxml2` options, Java `XMLInputFactory` with `XMLInputFactory.IS_SUPPORTING_EXTERNAL_ENTITIES=false`, etc.).
- **SSTI (Server-Side Template Injection)**: never pass user input as a template *source*; only as template *data*. Jinja, Freemarker, Velocity, Thymeleaf, Handlebars are all vulnerable to this pattern.
- **ReDoS**: avoid catastrophic regex (nested quantifiers like `(a+)+`). Limit regex input length. Prefer RE2 / linear-time regex engines for untrusted input.
- **Prototype pollution** (JS): use `Object.create(null)`, `Map`, or freeze prototypes; validate keys against `__proto__`, `constructor`, `prototype`.
- **Header injection**: reject CR/LF in values going into response headers, redirects, email headers.

## SSRF

When code fetches a URL provided by the user or derived from user data:
- Resolve the hostname and **reject private ranges** (10/8, 172.16/12, 192.168/16, 127/8, ::1, link-local `169.254/16`, cloud metadata `169.254.169.254`, IPv6 ULA `fc00::/7`).
- Force the scheme to `https://` (or an allow list of schemes; never `file://`, `gopher://`, `dict://`).
- **Cap response size and timeout** aggressively.
- **Disable redirects** or re-validate each redirect target with the same rules (DNS-rebinding defense: resolve once, connect to that IP).
- Consider a dedicated egress proxy for user-driven fetches.
- Beware DNS rebinding: resolution at validation time and resolution at connection time may differ.

## Open redirect

- Post-login, OAuth, "continue" redirects: validate the target URL against an **exact-match allow list** of paths or domains.
- Reject schemes other than relative paths or an approved `https://` host.
- Treat open redirect as a CSRF/phishing amplifier, not a nit.

## Path traversal

- `..`, absolute paths, symlinks out of the allowed root — all must be rejected when building paths from user input.
- Use `filepath.Clean` / `path.Clean` and then **verify the result is a descendant of the intended base directory** (compare canonical absolute paths).
- Never concatenate user input into file paths without validation.
- Windows: also reject `\`, drive letters, reserved names (`CON`, `PRN`, `AUX`, `NUL`).

## TOCTOU (Time-of-check to time-of-use)

- Do not `stat → check → open`. Do `open → check opened fd` (`O_NOFOLLOW`, verify inode).
- Do not rely on filename semantics after a race; use file descriptors.
- For uploads, move to final location only after validation is complete, atomically.

## Deserialization

- Reject unknown fields by default when deserializing configs or API bodies.
- **Never** deserialize pickled/serialized objects from untrusted sources: Python `pickle`, PHP `unserialize`, Java serialization, Ruby `Marshal.load`, YAML `!!python/object`.
- For YAML, use safe loaders only (`yaml.safe_load`).
- For JSON, reject unexpectedly large payloads before parsing; set parser limits.

## File uploads

- Validate MIME type **and** magic bytes, not just extension.
- Strip or reject archives that decompress beyond a limit (**zip-bomb** defense: ratio + absolute cap).
- Store uploads **outside the web root**; serve via a handler that enforces authz and `Content-Disposition`.
- For images, re-encode through a trusted library (stripping EXIF and preventing polyglot files).
- Randomize stored filenames; do not echo user-provided filenames back without sanitization.
- Antivirus scanning for user uploads if the product allows sharing.

## File downloads

- Sanitize `Content-Disposition` filenames (**reflected file download** attack): strip CR/LF, quotes, path separators; encode per RFC 6266.
- Never serve user-controlled HTML from the same origin as authenticated content. Use a sandboxed subdomain or `Content-Disposition: attachment`.
- Set `X-Content-Type-Options: nosniff`.

## Crypto

- **Do not roll your own.** Use vetted libraries: libsodium, BoringSSL, OpenSSL via a wrapper, Web Crypto, Go's `crypto` packages.
- **Random**: cryptographic RNG only (`crypto/rand`, `secrets.token_bytes`, `window.crypto.getRandomValues`). Never `Math.random` or `rand` for tokens, session IDs, password resets, invite codes.
- **Symmetric encryption**: AES‑GCM or ChaCha20‑Poly1305 (authenticated). Never ECB. Never AES‑CBC without a MAC.
- **Asymmetric**: Ed25519 for signatures (or RSA‑PSS if required), X25519 / ECDH P‑256 for key agreement. RSA < 2048 is dead.
- **KDF**: HKDF for key derivation from keying material; Argon2id/scrypt/PBKDF2 (≥ 600k iters) for passwords.
- **Nonces**: unique per key. GCM repeats → catastrophic. Use a counter or cryptographic RNG with large nonce.
- **TLS**: enforce TLS 1.2+, prefer TLS 1.3. Reject weak ciphers. HSTS on. No self-signed in prod.
- Don't implement custom MAC — use HMAC-SHA256 or Poly1305.

## Timing attacks & constant-time

- Comparing secrets (tokens, HMAC signatures, password hashes, reset tokens) MUST use constant-time comparison:
  - Go: `crypto/subtle.ConstantTimeCompare`
  - Python: `hmac.compare_digest`
  - Node: `crypto.timingSafeEqual`
  - Ruby: `Rack::Utils.secure_compare`
- Auth flows: equalize response time for "user exists / does not exist" and "wrong password / right password".
- Avoid early-return on secret-dependent branches.

## HTTP hardening

Response headers on HTML endpoints:
- `Strict-Transport-Security: max-age=31536000; includeSubDomains` (add `preload` when ready).
- `Content-Security-Policy` with explicit policy. **No `unsafe-inline`** in production; use nonces or hashes for required inline scripts. `default-src 'self'` as baseline.
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `X-Frame-Options: DENY` **or** CSP `frame-ancestors 'none'`.
- `Permissions-Policy` restricting features you do not use (camera, microphone, geolocation).
- `Cross-Origin-Opener-Policy: same-origin`, `Cross-Origin-Embedder-Policy: require-corp` for sensitive apps.

## CORS

- Do not reflect `Origin` blindly. Use an **allow list**.
- **Never** return `Access-Control-Allow-Origin: *` with `Access-Control-Allow-Credentials: true`.
- Preflight `OPTIONS` responses must be as strict as the real endpoint.
- For cookies: `SameSite=Lax` or `Strict` where possible; `Secure`; `HttpOnly` when the client does not need to read them.

## Rate limiting & abuse

- Rate limit authentication endpoints, password resets, email sends, webhook replays, and any expensive query.
- Keyed by user **and** by IP (fall back to IP when unauthenticated). Beware IP spoofing via `X-Forwarded-For` behind proxies; trust only your own edge.
- Consider CAPTCHA / proof-of-work / Turnstile on abuse-prone flows.
- Per-endpoint quotas for expensive operations (image resize, PDF gen, search, AI calls).
- Budget-level protection for external paid APIs (caps, alerts).

## Logging & audit

- Log **what happened, who did it, when, and the outcome**. Do not log secrets, passwords, session tokens, credit card numbers, or full request bodies on auth routes.
- **Redact PII** in logs by default; opt-in for detailed logs in dev only.
- **Log failed authorization attempts** — they are the primary signal of attack.
- Include a **correlation ID / trace ID** so a single request can be tracked end-to-end.
- **Immutable audit log** for sensitive actions: login, logout, role changes, data export, PII access, impersonation, permission grants. Append-only storage, ideally offsite.
- Monitor log ingestion itself — silent log drop is a detection gap.

## Error handling

- Server errors return a generic message to the client (`"internal error"` + a trace ID). Never return a stack trace, a SQL error, or an internal path to the client.
- Log the full error server-side with context.
- 4xx errors may describe the problem ("invalid email format"), but never reveal whether a user exists ("user not found" vs "wrong password" leaks account enumeration — use a single message).
- Fail closed on security-relevant errors (authz check errors ⇒ deny, not allow).

## Webhooks & callbacks

- **Incoming webhooks**: verify HMAC signature (vendor-provided secret) over the raw body; validate a **timestamp** within a small window (≤ 5 min) to prevent replay; store a nonce/id for idempotency.
- **Outgoing webhooks**: sign the payload, include a timestamp, use a dedicated URL-validated endpoint (SSRF rules apply), cap retries, back off exponentially.
- Always use TLS. Verify the target certificate.

## GraphQL

- Enforce **query depth** and **complexity** limits.
- Disable introspection in production for non-public schemas.
- Cap batched queries (`@batch`, array requests).
- Authz at the **resolver** level, not only at the gateway — every field.
- Persisted queries / allow list for public clients.

## WebSocket / SSE

- Validate `Origin` on upgrade; same allow list as CORS.
- Authenticate the connection with the same tokens as HTTP; re-check on privilege changes.
- Rate limit messages per connection; cap message size.

## Data at rest

- **Encrypt backups.** Restore tests run quarterly.
- Encryption at rest on the DB/disk when storing PII, PHI, PCI, or authentication secrets. Cloud-native (KMS-backed) is acceptable.
- **Field-level encryption** for highly sensitive columns (tokens, national IDs, health data) with a key managed in KMS/Vault.
- **Tokenize** credit card / national ID data when the product does not need the original value.

## Database & service accounts

- App connects with a **least-privileged DB user** (no `SUPERUSER`, no `CREATE/DROP DATABASE`). Separate users for migrations vs runtime.
- Separate read-only replicas for read workloads; app uses that user when possible.
- Every internal service has its own identity with scoped permissions. **No shared service accounts.**
- Rotate service credentials; prefer short-lived tokens (cloud IAM, Vault).

## Dependencies & supply chain

- Use a **lockfile** (`package-lock.json`, `yarn.lock`, `go.sum`, `poetry.lock`, `Cargo.lock`). Commit it.
- Run a vulnerability scanner in CI (`npm audit`, `pip-audit`, `govulncheck`, `cargo audit`, Trivy, Snyk, Dependabot/Renovate).
- Pin direct dependencies; update intentionally, not drive-by.
- Be suspicious of packages with few maintainers, recent ownership changes, or typo-squat-adjacent names.
- **Pin GitHub Actions by commit SHA**, not `@v4` (mutable tag can be repointed by a compromised maintainer).
- **Sign releases** (cosign / Sigstore). Verify signatures in the deploy pipeline.
- Publish an **SBOM** (CycloneDX or SPDX) on every release.
- Use SLSA-aware build systems where practical; at minimum, builds must be reproducible and run in ephemeral runners.
- Verify checksums of binaries downloaded at build time.

## Container & runtime baseline

- **Non-root user** in containers (`USER` directive). Drop Linux capabilities; set `readOnlyRootFilesystem: true` when possible.
- Never run with `--privileged` / `privileged: true`. Never mount `docker.sock` into containers that do not orchestrate.
- Minimal base images (distroless, Alpine, Wolfi). Scan images in registry.
- Pin base images by digest, not just tag.
- **Kubernetes**:
  - Enforce Pod Security Admission `restricted`.
  - Network policies default-deny egress and ingress; open explicitly.
  - RBAC: no `cluster-admin` for workloads. Service accounts per workload.
  - Secrets via CSI (Vault, Secrets Manager), not committed manifests.
  - `automountServiceAccountToken: false` unless needed.

## Infrastructure & IAM

- Least-privilege IAM roles. No wildcard `*:*`. No long-lived access keys — prefer OIDC / workload identity federation for CI to cloud.
- Separate accounts / projects for prod vs non-prod. No cross-boundary access without explicit policy.
- MFA required for human console access to cloud accounts.
- Enable cloud audit logs and centralize them (CloudTrail, Cloud Audit Logs, Activity Logs).
- Network segmentation: private subnets for DBs, public egress through NAT, security groups scoped per role.

## Test & staging data

- **Never use production data in dev or staging.** Seed with synthetic data or scrubbed/anonymized snapshots.
- Separate credentials, separate buckets, separate KMS keys between environments.
- Feature flags and debug endpoints must be off-by-default in production.
- Disable test-only backdoors before shipping; CI should grep for them.

## Monitoring & alerting

- Alert on: spikes in 401/403, new IP admin login, privilege grant, secret rotation failure, cert expiration approaching, anomalous data export volume.
- Security dashboards reviewed on a cadence; not passive logs.
- Incident response runbook exists; on-call knows where to find it.

## AI-specific

When the product calls LLMs on behalf of users:
- **Never** include unredacted PII, secrets, or proprietary data in the prompt.
- Treat LLM **output as untrusted input**. Do not execute it as code, SQL, shell, or tool calls without the same validations applied to user input.
- **Prompt injection** is real. If the model reads from user-controlled content (documents, emails, web pages, tool results), treat that content as adversarial and sandbox any resulting actions (no arbitrary tool access, allow lists, human-in-the-loop for destructive ops).
- Cap tokens per request and budget per user to prevent cost-exfiltration abuse.
- Log prompts and responses with PII redaction; respect data-residency constraints.
- Do not store user prompts for training without explicit, informed consent.

## Before marking Ready (checklist)

For each PR that touches a sensitive surface, answer:

- [ ] Any new user input goes through a validator (type, length, range, encoding)?
- [ ] Authorization checked for every new endpoint / action (including `GET`)?
- [ ] Multi-tenant isolation verified (resource ownership, not just existence)?
- [ ] Any new SQL / shell / template / regex use is parameterized, escaped, or safe?
- [ ] Any outbound fetch from user input has SSRF protection (private-range reject, size/timeout caps)?
- [ ] Any redirect target from user input is allow-listed?
- [ ] Any file path built from user input is confined to a safe root?
- [ ] Any secret added to env/vault (not committed), not logged, not in URL?
- [ ] Any new crypto uses a vetted library and a correct algorithm (not DIY)?
- [ ] Constant-time comparison for secret equality?
- [ ] State-changing cookie-auth endpoints protected against CSRF?
- [ ] New session / token flow rotates on login, invalidates on logout, uses httpOnly cookies?
- [ ] Error responses do not leak stack traces, SQL, or internal paths?
- [ ] Rate limits applied to any expensive or abuse-prone endpoint?
- [ ] Logs redact PII and secrets?
- [ ] Sensitive actions recorded in the audit log?
- [ ] Dependency additions scanned for CVEs; actions pinned by SHA?
- [ ] Container/K8s changes respect non-root, least-privilege, network policy defaults?
- [ ] No prod data introduced into dev/staging?

If any box is "no" and that surface exists, it is a blocker, not a nit.

## Forbidden

- Disabling TLS verification in code (`verify=False`, `InsecureSkipVerify`) outside of clearly scoped local development.
- Hardcoded secrets anywhere, even "temporary".
- `eval`, `Function("…")`, `exec` on strings built from user input.
- Writing custom cryptography (rolling your own KDF, MAC, cipher, or RNG).
- Catching a security-relevant error and continuing silently.
- Disabling a lint/scanner rule without a recorded justification.
- `SELECT *` with user input concatenated. Even once. Even "just for now".
- Storing tokens or credentials in `localStorage` / `sessionStorage`.
- Committing `.env`, `*.pem`, `credentials.*`, or kube-config files.
- Running containers as root or with `--privileged` in any shared environment.
- `SameSite=None` without `Secure`.
- Dev-only auth bypasses (`?admin=true`, `X-Dev-Role: admin`) that can reach production builds.
- Swallowing CSP/SRI errors; "just turn it off" is the wrong fix.
